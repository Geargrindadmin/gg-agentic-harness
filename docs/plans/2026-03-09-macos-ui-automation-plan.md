# macOS Control Surface UI Automation Plan

Date: 2026-03-09
Status: in progress
Owner: harness integration

## Goal

Move the macOS control surface from build-and-manual validation to deterministic automated coverage for the operator-critical flows:

- planner task creation and launch
- coordinator and sub-agent runtime selection
- swarm telemetry visibility
- worker steering
- worktree inspection
- agent analytics visibility

## Current State

What exists now:

- the mac app builds and launches
- the control plane exposes planner, swarm, worktree, usage, and analytics endpoints
- the app has shared planner/swarm/console context
- the app now has a first Swift test target with fixture-backed client, workflow-context, and runtime-setting tests

What is missing:

- deterministic UI-flow coverage for planner and swarm views
- fixture-driven testing for view models beyond the REST client
- launch-time app smoke coverage against a stable local fixture backend
- regression coverage for typing/focus, worker steering, and worktree presentation

## Delivery Phases

### Phase 1 — Deterministic Transport Seam

Done:

- add a mac app Swift test target
- add fixture-backed `A2AClient` transport override
- add first tests for:
  - `fetchAgentAnalytics`
  - `fetchBusStatus`
  - shared `WorkflowContextStore` selection/sync behavior
  - `AgentSwarmModel` bus ingestion + communication-edge rendering
  - `WorktreeViewModel` success + missing-worktree handling
  - `PlannerStore` reload + create-task refresh flow
  - generic sub-agent runtime selection logic

### Phase 2 — Fixture Control-Plane Dataset

In progress:

  - run list

- make fixture responses represent:
  - codex coordinator + kimi workers
  - claude coordinator + codex workers

### Phase 3 — View Model Coverage

In progress:

Done:

### Phase 4 — App Flow Coverage

Next:

- add app-level smoke coverage for:
  - planner launch state
  - task selection propagating to swarm/console
  - selected worker telemetry visibility
  - worktree panel open path

Preferred approach:

- start with deterministic state tests and app-process smoke tests
- only add full XCUITest once the app is wrapped in an Xcode UI-testable bundle or project setup

### Phase 5 — Operator Regression Pack

Next:

- cover:
  - typing/focus in planner text editors
  - coordinator switching
  - sub-agent runtime switching
  - worker retry / retask / terminate actions
  - agent analytics rendering with non-empty data

## Acceptance Criteria

This slice is complete when:

1. the app test target runs in CI/local without a live harness session
2. fixture-backed tests cover planner/swarm/analytics decoding and state transitions
3. worktree and telemetry regressions are caught by tests before release
4. manual-only validation is no longer the primary verification path for core operator flows

## Notes

- keep the control plane headless-first; tests should not require the mac app to launch external CLIs
- prefer fixture data that mirrors real run-state JSON generated by the harness
- keep UI automation focused on operator-critical paths, not every decorative surface
- fixture-backed suites touching the shared `A2AClient` singleton are now gated to avoid cross-suite override races
