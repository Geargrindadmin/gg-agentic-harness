Operational Flow

The harness still follows the same five-step shape when the app is absent. The app just makes these settings visible and editable.
Stage 1

Intake and Priming

  • Request is classified and normalized.
  • Runtime profile, project context, and memory are loaded.
  • Prompt improver and context source policy shape the run.
Stage 2

Governance and Routing

  • Board path activates for high-risk or architecture work.
  • Persona routing resolves coordinator and worker roles.
  • Dispatch plan decides runtime mix and worker fanout.
Stage 3

Execution Engine

  • Coordinator launches the swarm and owns the run graph.
  • Workers inherit a harness system prompt plus active policy.
  • Loop budget and retry budget are included in that contract.
Stage 4

Verification and Docs

  • Validation mode controls which checks are expected.
  • Doc sync mode controls whether docs are expected inline.
  • Results are written to run state, logs, and artifacts.
Stage 5

Feedback Hardening

  • Recurring failures become evidence, not silent retries.
  • Governor limits shape safe worker capacity.
  • Portable installs can verify parity and settings headlessly.
