GG Agentic Harness - Multi-Model Control Plane

Placement proposal for harness-native orchestration across codex, claude, and kimi. The control plane lives inside Node 3.5, between runtime selection and the 5-Cycle execution engine.

PRD: docs/prd/PRD-MULTI-MODEL-CONTROL-PLANE.md Placement: Node 3.5 control plane Authority: harness-owned spawn and delegation Messaging: mailbox with ack, not peer chat Adapters: codex | claude | kimi

Control-Plane Placement

Placement summary: multi-model orchestration becomes a harness-core control plane at Node 3.5, not an external peer-chat bridge.

Node 1 Entry Points

The top-left entry boxes show CLI, IDE, and workflow entry into the harness.

Node 2 Agent Sandbox

The adapter layer is wrapped in sandboxed worker execution rather than direct peer control.

Node 3A / 3B Active Coordinator Runtime

The selected runtime acts as coordinator mode before any delegation happens.

Node 3.5 Multi-Model Control Plane

This is the new placement: run registry, delegation policy, mailbox bus, and worker lifecycle.

Node 4 Blueprint Engine

The 5-Cycle engine remains the execution contract after worker routing is resolved.

Node 5 Rules and Context

Classification, governance, and fail-closed policy remain central harness responsibilities.

Node 5.5 Memory Layer

The context-prime block shows the memory and project-context path used before dispatch.

Node 6 Tool Shed

Persona routing, capability checks, mailbox tools, and adapter capabilities sit in the tool layer.

Node 7 Validation Layer

Delegated work still returns through deterministic validation gates owned by the harness.

Node 8 PR Review and Ship

The ship path remains the final output lane after validation and documentation sync.

Why This Fit Is Clean

Preserves harness authority.
Validation, governance, and run artifacts stay in one place.
Avoids peer-chat drift.
Workers use inbox/outbox semantics instead of natural-language backchannels.
Respects runtime parity.
`codex`, `claude`, and `kimi` remain profile-driven with adapter capability checks.
Supports model specialization.
Codex can coordinate, Claude can plan/review, Kimi can build in parallel.

First CLI Surface

Command Purpose
gg run create Initialize a multi-model run and return runId.
gg worker spawn --runtime codex Spawn a harness-managed worker through a runtime adapter.
gg worker delegate --to-runtime kimi Submit a policy-evaluated delegation request.
gg bus inbox --agent-id builder-1 Read mailbox entries for one worker.
gg run watch <runId> Track worker status, health, and completion.
